Some key improvements are:</p>\n<h3>Disabling Literals</h3>\n<p>Literals can be disabled in <a href=\"http://docs.phalconphp.com/en/latest/reference/phql.html\">PHQL</a>, this means that directly using strings, numbers and boolean values in PHQL strings will be disallowed. In Phalcon 1.3 this option has been improved and now works with<br/>most query builders, finders and PHQL generators.</p>\n<pre class=\"sh_php sh_sourceCode\">Phalcon\\Mvc\\Model::setup(array('phqlLiterals' =&gt; false));\n\n$phql = \"SELECT Robots.* FROM Robots WHERE Robots.type = :type: LIMIT :limit:\";\n\n$result = $this-&gt;modelsManager-&gt;executeQuery($phql, array(\n\t'type' =&gt; $this-&gt;request-&gt;getPost('name')\n\t'limit' =&gt; $this-&gt;request-&gt;getPost('limit')\n));\n</pre>\n<h3>Registry</h3>\n<p>A widely requested by the community component, Phalcon\\Registry is introduced in this version. This component implements the Registry pattern, allowing the developer to store and retrieve all kinds of values using simple keys.</p>\n<pre class=\"sh_php sh_sourceCode\">\n$registry = new \\Phalcon\\Registry();\n\n//Store a simple value\n$registry-&gt;mydata = \"hello\";\n\n//Store a complex resultset\n$registry-&gt;robots = Robots::find();\n</pre>\n<p>Registries can be iterated as arrays:</p>\n<pre class=\"sh_php sh_sourceCode\">foreach ($registry as $key =&gt; $value) {\n\tvar_dump($key);\n\tvar_dump($value);\n}\n</pre>\n<p>And can be accessed as objects or arrays:</p>\n<pre class=\"sh_php sh_sourceCode\">echo $registry-&gt;key;\necho $registry['some key'];\necho $registry[$key];\necho $registry-&gt;{$key};\necho $registry[0];\necho $registry-&gt;{0};\n</pre>\n<h3>Ini Settings</h3>\n<p>Phalcon 1.3 introduces php.ini settings to alter the global behavior of the extension:</p>\n<pre class=\"sh_php sh_sourceCode\">; Enables/Disables globally the internal events\nphalcon.orm.events = On\n\n; Enables/Disables virtual foreign keys\nphalcon.orm.virtual_foreign_keys = On\n\n; Enables/Disables column renaming\nphalcon.orm.column_renaming = On\n\n; Enables/Disables automatic NOT NULL validation\nphalcon.orm.not_null_validations = On\n\n; Enables/Disables throwing an exception if save fails\nphalcon.orm.exception_on_failed_save = On\n\n; Enables/Disables literals in PHQL\nphalcon.orm.enable_literals = On\n\n; Enables/Disables auttomatic escape\nphalcon.db.escape_identifiers = On\n</pre>\n<h3>PSR-3 Logger Implementation</h3>\n<p>Phalcon\\Logger now implements the <a href=\"http://www.php-fig.org/psr/psr-3/\">PSR-3</a> standard for loggers which allows this component interoperate with other frameworks that also implement this standard.</p>\n<p>In addition to this, Phalcon optionally provides C implementations of the interfaces and traits required by this standard as part of the extension:</p>\n<ul><li><span>Psr\\Log\\LoggerInterface,</span></li>\n<li><span>Psr\\Log\\LoggerAwareInterface,</span></li>\n<li><span>Psr\\Log\\LogLevel,</span></li>\n<li><span>Psr\\Log\\LoggerAwareTrait and</span></li>\n<li><span>Psr\\Log\\LoggerTrait.</span></li>\n</ul><p>You can enable/disable them via the following php.ini directive:</p>\n<pre class=\"sh_php sh_sourceCode\">phalcon.register_psr3_classes = On;\n</pre>\n<p>This also would allow other components implement and use these interfaces directly where Phalcon is installed.</p>\n<h3>Image Manipulation Component</h3>\n<p>A component that was conceived and implemented by our wonderful contributors Phalcon\\Image is available for developers that need to manipulate images using GD or Imagemagick. A unified interface is provided to manipulate images using either library.</p>\n<pre class=\"sh_php sh_sourceCode\">use Phalcon\\Image\\Adapter\\GD as GdAdapter;\nuse Phalcon\\Image\\Adapter\\Imagick as ImagickAdapter;\n\n$image = new GdAdapter('my-logo.jpg');\n\n$image\n\t-&gt;crop(200, 200)\n\t-&gt;resize(70, 50)\n\t-&gt;save('new-logo.jpg');\n</pre>\n<p>For now only the most commonly used functions have been unified and implemented since both libraries provide different functionality.</p>\n<h3>Performance Improvements</h3>\n<p><strong>TLDR:</strong> Phalcon is now faster!</p>\n<p>The following performance improvements have been implemented in this version, they are a bit technical, you can skip them if you want:</p>\n<p><strong>Polymorphic Function/Method cache</strong><br/>Functions and methods are called directly by Phalcon using C calls taking advantage that compilers know their address/location at compile time, however, others only exist temporarily at runtime and must be resolved by Phalcon in the PHP userland. Version 1.3 has introduced a cache that remembers (when possible) the location of these very dynamic functions and methods and reuse those locations in subsequent calls to the same method/function in order to improve performance.</p>\n<p><strong>Global constants</strong><br/>Some simple and constant values such as: null, false, true, 1, 0 and others, are often used by Phalcon throughout its execution cycle. Previously, these values were (in some cases) allocated or duplicated unnecessarily. Phalcon 1.3 just made one allocation per request for these values and then they are shared everywhere where they are used reducing overall memory usage.</p>\n<p><strong>Memory Management</strong><br/>A global memory tracker is used by Phalcon to track every segment of memory allocated by the framework in order to accordingly release and free that memory when required. Earlier versions of Phalcon needed to request additional memory to track those memory segment. Phalcon 1.3 does not need to do this in most execution cycles, thus improving performance.</p>\n<p><strong>Internal Properties</strong><br/>Many of the properties previously implemented classes were Phalcon PHP userland properties. Now many of them have been rewritten to take advantage of internal properties that have a much lower overhead when reading and writing. For example, Phalcon does not need to check whether the visibility of these properties is public, protected or private, the framework knows exactly whether they can be read or updated.</p>\n<p><strong>Hidden Symbols</strong> <br/>Most of the symbols and names of internal functions and structures are not exported publicly anymore allowing the compiler to perform more aggressive optimizations by knowing in advance the address of functions and knowing that those will not be overridden externally.</p>\n<h3>Acknowledgments</h3>\n<p>Big kudos to main contributors on Version 1.3 has introduced a cache that remembers (when possible) the location of these very dynamic functions and methods and reuse those locations in subsequent calls to the same method/function in order to improve performance.</p>\n<p><strong>Global constants</strong><br>Some simple and constant values such as: null, false, true, 1, 0 and others, are often used by Phalcon throughout its execution cycle. Earlier versions of Phalcon needed to request additional memory to track those memory segment. Phalcon 1.3 does not need to do this in most execution cycles, thus improving performance.</p>\n<p><strong>Internal Properties</strong><br>Many of the properties previously implemented classes were Phalcon PHP userland properties. Now many of them have been rewritten to take advantage of internal properties that have a much lower overhead when reading and writing. updated.</p>\n<p><strong>Hidden Symbols</strong> <br>Most of the symbols and names of internal functions and structures are not exported publicly anymore allowing the compiler to perform more Right after our [2 year
celebration](http://blog.phalconphp.com/post/78883849784/3acec916ae9c026594fe0562ef3013be2cf47247)
and several months of development, more than 1,500 commits and a lot of
improvements and fixes, **Phalcon 1.3 is finally released**!

Many important internal aspects of Phalcon have been rewritten and
enhanced without breaking compatibility with older versions.

Phalcon 1.3 consolidates the research, testing and implementation of new
ways to improve the performance in the Phalcon kernel, which is also the
heart of the extensions created by
[Zephir](http://blog.zephir-lang.com/) in Phalcon 2.0.

The
[CHANGELOG](https://github.com/phalcon/cphalcon/blob/master/CHANGELOG)
for this version is huge. Some key improvements are:

### Disabling Literals

Literals can be disabled in
[PHQL](http://docs.phalconphp.com/en/latest/reference/phql.html), this
means that directly using strings, numbers and boolean values in PHQL
strings will be disallowed. In Phalcon 1.3 this option has been improved
and now works with\
most query builders, finders and PHQL generators.

```php
Phalcon\Mvc\Model::setup(array('phqlLiterals' => false));

$phql = "SELECT Robots.* FROM Robots WHERE Robots.type = :type: LIMIT :limit:";

$result = $this->modelsManager->executeQuery($phql, array(
    'type' => $this->request->getPost('name')
    'limit' => $this->request->getPost('limit')
));
```

### Registry

A widely requested by the community component, Phalcon\\Registry is
introduced in this version. This component implements the Registry
pattern, allowing the developer to store and retrieve all kinds of
values using simple keys.

```php
$registry = new \Phalcon\Registry();

//Store a simple value
$registry->mydata = "hello";

//Store a complex resultset
$registry->robots = Robots::find();
```

Registries can be iterated as arrays:

```php
foreach ($registry as $key => $value) {
    var_dump($key);
    var_dump($value);
}
```

And can be accessed as objects or arrays:

```php
echo $registry->key;
echo $registry['some key'];
echo $registry[$key];
echo $registry->{$key};
echo $registry[0];
echo $registry->{0};
```

### Ini Settings

Phalcon 1.3 introduces php.ini settings to alter the global behavior of
the extension:

```php
; Enables/Disables globally the internal events
phalcon.orm.events = On

; Enables/Disables virtual foreign keys
phalcon.orm.virtual_foreign_keys = On

; Enables/Disables column renaming
phalcon.orm.column_renaming = On

; Enables/Disables automatic NOT NULL validation
phalcon.orm.not_null_validations = On

; Enables/Disables throwing an exception if save fails
phalcon.orm.exception_on_failed_save = On

; Enables/Disables literals in PHQL
phalcon.orm.enable_literals = On

; Enables/Disables auttomatic escape
phalcon.db.escape_identifiers = On
```

### PSR-3 Logger Implementation

Phalcon\\Logger now implements the
[PSR-3](http://www.php-fig.org/psr/psr-3/) standard for loggers which
allows this component interoperate with other frameworks that also
implement this standard.

In addition to this, Phalcon optionally provides C implementations of
the interfaces and traits required by this standard as part of the
extension:

-   Psr\\Log\\LoggerInterface,
-   Psr\\Log\\LoggerAwareInterface,
-   Psr\\Log\\LogLevel,
-   Psr\\Log\\LoggerAwareTrait and
-   Psr\\Log\\LoggerTrait.

You can enable/disable them via the following php.ini directive:

```php
phalcon.register_psr3_classes = On;
```

This also would allow other components implement and use these
interfaces directly where Phalcon is installed.

### Image Manipulation Component

A component that was conceived and implemented by our wonderful
contributors Phalcon\\Image is available for developers that need to
manipulate images using GD or Imagemagick. A unified interface is
provided to manipulate images using either library.

```php
use Phalcon\Image\Adapter\GD as GdAdapter;
use Phalcon\Image\Adapter\Imagick as ImagickAdapter;

$image = new GdAdapter('my-logo.jpg');

$image
    ->crop(200, 200)
    ->resize(70, 50)
    ->save('new-logo.jpg');
```

For now only the most commonly used functions have been unified and
implemented since both libraries provide different functionality.

### Performance Improvements

**TLDR:** Phalcon is now faster!

The following performance improvements have been implemented in this
version, they are a bit technical, you can skip them if you want:

**Polymorphic Function/Method cache**\
Functions and methods are called directly by Phalcon using C calls
taking advantage that compilers know their address/location at compile
time, however, others only exist temporarily at runtime and must be
resolved by Phalcon in the PHP userland. Version 1.3 has introduced a
cache that remembers (when possible) the location of these very dynamic
functions and methods and reuse those locations in subsequent calls to
the same method/function in order to improve performance.

**Global constants**\
Some simple and constant values such as: null, false, true, 1, 0 and
others, are often used by Phalcon throughout its execution cycle.
Previously, these values were (in some cases) allocated or duplicated
unnecessarily. Phalcon 1.3 just made one allocation per request for
these values and then they are shared everywhere where they are used
reducing overall memory usage.

**Memory Management**\
A global memory tracker is used by Phalcon to track every segment of
memory allocated by the framework in order to accordingly release and
free that memory when required. Earlier versions of Phalcon needed to
request additional memory to track those memory segment. Phalcon 1.3
does not need to do this in most execution cycles, thus improving
performance.

**Internal Properties**\
Many of the properties previously implemented classes were Phalcon PHP
userland properties. Now many of them have been rewritten to take
advantage of internal properties that have a much lower overhead when
reading and writing. For example, Phalcon does not need to check whether
the visibility of these properties is public, protected or private, the
framework knows exactly whether they can be read or updated.

**Hidden Symbols** \
Most of the symbols and names of internal functions and structures are
not exported publicly anymore allowing the compiler to perform more
aggressive optimizations by knowing in advance the address of functions
and knowing that those will not be overridden externally.
